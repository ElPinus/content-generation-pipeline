# Architecture Notes — Content Generation Pipeline

> **Дисклеймер.** Это публичное описание архитектуры реальной системы, в разработке которой автор участвовал. Конкретные клиенты, доменные имена, финансовые показатели, исходный код и проприетарные детали реализации не раскрываются. Содержание ограничено архитектурными решениями и принципами, обсуждаемыми в публичном поле для систем такого назначения.

Описание ключевых архитектурных решений: контекст (что решали) и реализация (как сделано).

---

## 1 · Стадии как переиспользуемые компоненты

### Контекст

В системе несколько сценариев контент-генерации (длинные нарративные посты, короткие посты с медиа, серии под отдельные кампании). Некоторые шаги повторяются между сценариями, некоторые отличаются.

### Реализация

Stage — модуль с явными input/output контрактами и handler'ом. Pipeline — упорядоченный список ссылок на стадии с их конфигом, хранится в БД как шаблон. Передача данных между стадиями — через сериализованный JSON-context, накапливаемый по мере прохождения. Одна реализация стадии используется в нескольких сценариях; новый сценарий собирается комбинацией существующих стадий, без изменения кода.

---

## 2 · Конструктор пайплайнов в UI

### Контекст

Целевой пользователь — продакт или контент-стратег, который лучше всех знает какой пайплайн нужен для конкретного направления, но кода не пишет.

### Реализация

Стадии берутся из реестра, зарегистрированного backend'ом; пользователь располагает их в нужном порядке через визуальный конструктор; для каждой стадии — динамическая форма конфигурации, поля которой соответствуют её config_schema. Готовый пайплайн сохраняется в БД как шаблон и может быть запущен на конкретном брифе.

---

## 3 · Human-in-the-loop как полноценная стадия

### Контекст

LLM на текущем уровне не даёт стабильной гарантии качества: может выдать общий, повторяющийся или несоответствующий бренду текст. В большинстве реальных сценариев необходима точка ручного контроля.

### Реализация

Manual-approval / manual-edit — типы стадий, как и любые автоматические. При входе в такую стадию pipeline переходит в состояние `awaiting_approval` и останавливается; возобновление — только по явному действию пользователя (одобрить / редактировать / отклонить).

Pipeline state-machine имеет состояние `AwaitingApproval`; отдельный approval-сервис регистрирует pending-action и оповещает пользователя; UI собирает очередь approval'ов; действие пользователя обновляет stage_execution и enqueue'ит следующую стадию. Авто и ручные стадии можно произвольно перемешивать.

## 4 · Состояние пайплайна — в основной реляционной БД

### Контекст

Каждый запуск пайплайна (pipeline_run) проходит через несколько стадий, каждая из которых меняет статус и накапливает контекст. Это «второй» вид данных, идущий рядом с бизнес-данными (шаблонами, контентом, публикациями).

### Реализация

Шаблоны, runs и stage_executions живут в одной PostgreSQL с контентом и публикациями. Завершение стадии и фиксация её результата в бизнес-таблицах выполняются в одной транзакции БД. Один контур наблюдения — одна БД, один admin-инструмент, один бэкап.

---

## 5 · Хранение credentials в БД с прозрачным шифрованием на уровне ORM

### Контекст

Пользователи привязывают к проектам credentials двух типов: API-ключи LLM-провайдеров и OAuth-токены социальных платформ. Первое — биллинговая поверхность; второе — доступ к корпоративным аккаунтам клиента.

### Реализация

Custom-тип колонки на уровне ORM (`EncryptedText`) автоматически шифрует значение Fernet'ом при записи и расшифровывает при чтении в Python-процессе. Encryption key хранится в env, не в БД. В UI значения секретов всегда маскированы; полный текст видим только в момент создания/редактирования. Бэкапы и физические дампы БД не содержат plaintext.

---

## 6 · Многоканальное расписание публикаций

### Контекст

Один контент-результат часто публикуется в несколько каналов в разное время — например, пост в платформе A в 10:00, его адаптированная версия в платформе B в 14:00.

### Реализация

Запланированная публикация — это запись в БД со связями на контент-результат и интеграцию с платформой, поле `scheduled_at` и статусная машина (`scheduled` → `publishing` → `published` / `failed`). Периодическая задача (Celery beat) с малым шагом ищет публикации, у которых пришло время, переводит их в `publishing`, передаёт на воркер. Перенос публикации — обновление `scheduled_at`. В UI всегда видно, что и когда будет опубликовано.
